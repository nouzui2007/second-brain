# mail-to-todo

このスキルは、メールボックスを確認して「実行すべきタスク」を抽出し、
このリポジトリの `todo/` に登録した上で、確認用のブランチ・Pull Requestを作成します。
メール本文は誤読・誤分類のリスクがあるため、**mainには直接pushせず、必ずブランチ+PR経由**にします。

## 対象範囲

- 対象はメールのみ(チャット入力・画像からの整理は `organize-notes` スキルの役割)。
- 対象メールボックス: (ユーザーが指定するアカウント。未指定なら接続されているGmailのメインアカウントを使う)
- 確認範囲: 前回チェック以降の新着メール、または直近48時間の未読メール(明示指定があればそれに従う)。

## 実行手順

### 1. 新着メールの取得

1. Gmail連携(またはメール用MCPツール)で、未読メール一覧を取得する。
2. すでに処理済みのメールを再処理しないよう、処理済みメールには `処理済み: mail-to-todo` のラベルを付与し、次回以降はこのラベルが付いたメールをスキップする。
3. ラベルが使えない場合は、`todo/` 側に記録した「元メールのMessage-ID」を突き合わせて重複登録を避ける。

### 2. タスク判定・抽出

各メールについて、以下を確認する。

- 依頼・締め切り・確認事項などタスク性のある内容が含まれるか判定する。
  - 判定基準の例: 「〜してください」「〜までにご確認を」「至急」「〆切」「対応お願いします」等の表現、日付を伴う依頼。
  - 単なる情報共有・広告・通知メールはタスクとして扱わない。
- タスクと判定したメールから以下を抽出する:
  - **タイトル**: 件名または本文から要約した短い名前
  - **概要**: 依頼内容の要約(誰から・何を・なぜ)、元メールの件名・送信者・受信日時
  - **開始日**: メール受信日、または本文中に明記があればその日付
  - **終了日 / 期限**: 本文中の期限表現から抽出。読み取れなければ「未定」とし、本文中の該当箇所を概要に引用して人間が確認できるようにする。
- どのプロジェクトに属するか、送信者・件名・本文のキーワードから推定する。
  - 推定できる場合: `todo/<プロジェクト名>.md` に登録
  - 推定できない場合: `todo/inbox.md` に登録し、概要に「要分類」と明記する

### 3. 書き込みルール(todo/との整合)

`organize-notes` スキルの `todo/` ルールに従うこと。

- タスクは1件1見出しで追記する。
- フォーマット:

  ```markdown
  ### <タイトル>
  - 概要: <要約>(元メール: 件名「〜」、送信者: 〜、受信日: 2026-07-11)
  - 開始日: <日付 or 未定>
  - 終了日: <日付 or 未定>
  - ステータス: 未着手
  ```

- 既存ファイルは追記前に必ず読み込み、同一メール由来のタスクが既に存在しないか確認する。

### 4. ブランチ作成・コミット・push・PR作成

1. 作業用ブランチを作成する: `mail-to-todo/YYYY-MM-DD` (同日に複数回実行する場合は連番を付ける)
2. 変更したtodoファイルをコミットする。コミットメッセージ例:
   `mail-to-todo: 新着メール3件からTODOを追加`
3. ブランチをリモートにpushする。
4. Pull Requestを作成する。PR本文には以下を含める:
   - 今回処理したメールの件名一覧(送信者・受信日時つき)
   - 追加/更新したtodoファイルとタスクの一覧
   - 「要分類」「期限不明」など、人間の確認が必要な項目のハイライト
5. **mainブランチへの直接commit・pushは行わない。** ユーザーがPRを確認しマージするまでは反映されない状態を維持する。

### 5. 完了報告

作業完了後、チャット上で以下を簡潔に報告する。

- 処理したメール件数、うちタスクと判定した件数
- 作成したPRのリンク(または該当ブランチ名)
- 確認が必要な項目(期限不明・プロジェクト不明など)

## 注意事項

- メール本文の内容をそのままtodoファイルに転記しない。要約・整形した上で記載する。
- 個人情報や機密情報を含むメールは、概要に必要最小限の情報のみを残す(本文全文の引用はしない)。
- 誤判定が疑われる場合(スパム、営業メール等)は無理にタスク化せず、スキップしてよい。判断に迷う場合はPR本文に「タスク化を保留したメール」として一覧を added すると親切。
- 定期実行(スケジュール)で動かす場合も、push先は必ずブランチ+PRとし、mainへの直接反映は行わない。
