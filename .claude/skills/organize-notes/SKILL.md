# organize-notes

このリポジトリは個人のメモとTODOをMarkdownで管理するためのものです。
ユーザーからテキスト・画像(カレンダー、WBS、スケジュール表など)が渡されたら、
以下のルールに従って `memo/` と `todo/` に情報を振り分けてください。

## 基本方針

1. 入力(自然言語 / 画像)から「純粋な記録・知識・気づき」と「実行すべきタスク」を分離する。
2. 前者は `memo/` へ、後者は `todo/` へ書く。
3. **メモの中にTODOが含まれる場合は、memoファイルに記載した上で、該当タスクを todo/ 側にも重複して登録する。**
   - memo側には元の文脈を残し、todo側には実行管理に必要な情報だけを整形して書く。
4. 既存ファイルがあれば追記、なければ新規作成する。作業前に対象ファイルを必ず読み込み、重複登録を避ける(同一タイトル・同一期日のタスクが既にないか確認)。

## memo/ のルール

- ファイルはテーマ単位で分割する(例: `仕事.md`, `健康.md`, `学習.md`, `家族.md`)。
- どのテーマにも当てはまらない場合はユーザーに一言確認するか、`雑記.md` に入れる。
- 各エントリは日付付きの見出しか箇条書きで追記する。フォーマット例:

  ```markdown
  ## 2026-07-11
  - 〇〇プロジェクトの要件定義MTGで、△△の仕様が未確定という話が出た。
  - 次回MTGまでに□□さんに確認する必要あり。 → todo/project-a.md に追加済み
  ```

## todo/ のルール

- ファイルは **プロジェクト単位** で分割する(例: `project-a.md`)。
- プロジェクトが不明・未分類の場合は `inbox.md` に一旦入れ、後で分類する。
- 各タスクは必ず以下の項目を持つこと:
  - **タイトル**: 一目でわかる短い名前
  - **概要**: 何をするか、背景や参照元メモへのリンク
  - **開始日**: 分かれば記載、不明なら「未定」
  - **終了日 / 期限**: 分かれば記載、不明なら「未定」
  - **ステータス**: 未着手 / 進行中 / 完了 のいずれか(デフォルトは未着手)
- フォーマット例(1タスク1見出し):

  ```markdown
  ### 仕様確認を□□さんに依頼する
  - 概要: 要件定義MTGで未確定だった△△の仕様について確認する。(元メモ: memo/仕事.md 2026-07-11)
  - 開始日: 2026-07-11
  - 終了日: 2026-07-18
  - ステータス: 未着手
  ```

- 完了したタスクは削除せず「ステータス: 完了」に更新する(履歴として残す)。

## 画像入力(カレンダー・WBS)の扱い

- カレンダー画像: 写っている予定を日付・タイトル単位で読み取り、各予定を対応するプロジェクトのtodoファイルにタスクとして登録する。プロジェクトが読み取れない予定は `inbox.md` へ。開始日・終了日は画像上の日付をそのまま使う(終日予定は開始日=終了日でよい)。
- WBS画像: 各行(タスク)をタイトル・開始日・終了日として抽出し、WBSが属するプロジェクトのtodoファイル(`todo/<プロジェクト名>.md`)にまとめて追記する。親子関係(大項目・中項目)がある場合は見出しレベルで表現する。
- 画像の読み取り結果は必ず一度テキストで書き出し、ユーザーに「この内容で登録します」と確認してから書き込む(誤読による誤登録を防ぐため)。

## 実行手順

1. 入力内容を確認し、メモ的要素とタスク的要素に分ける。
2. 関連する既存ファイル(memo/todoの両方)を読み、重複や関連エントリがないか確認する。
3. memo側に追記する内容を作成する。
4. TODOが含まれる場合は、対応するtodoファイルにタスクを追記する(なければ新規作成)。
5. 変更したファイル一覧と要約を最後に提示する。
6. 特に指示がなければ、そのままgit commitはせず、ユーザーの確認を待つ。
